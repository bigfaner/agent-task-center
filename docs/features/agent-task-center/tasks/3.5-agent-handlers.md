# 3.5: Agent HTTP handlers

## Description

在 `internal/handler/` 实现所有 Agent 端点的 HTTP handler，挂载到 `/api/agent/` 前缀。

**需实现的端点：**

| Method | Path | Handler |
|--------|------|---------|
| POST | `/api/agent/claim` | `ClaimTask` |
| PATCH | `/api/agent/tasks/{taskId}/status` | `UpdateTaskStatus` |
| POST | `/api/agent/tasks/{taskId}/records` | `SubmitRecord` |
| GET | `/api/agent/tasks/{taskId}/content` | `GetTaskContent` |

**ClaimTask 请求体：**
```json
{ "projectName": "...", "featureSlug": "...", "agentId": "..." }
```

**UpdateTaskStatus 请求体：**
```json
{ "agentId": "...", "status": "blocked" }
```

**SubmitRecord 请求体：** 见 api-handbook，包含 agentId + 所有执行记录字段。

**GetTaskContent：** `taskId` 为字符串格式（如 `1.2`），从 query string 读取 `project` 和 `feature`。

注意：`/api/agent/tasks/{taskId}` 中的 `taskId` 路径参数在 claim/record 端点为数字 ID，在 content 端点为字符串 task_id（如 `1.2`）。

## Acceptance Criteria

- [ ] `POST /api/agent/claim` 无可用任务时返回 404 `no_available_task`
- [ ] `PATCH /api/agent/tasks/{id}/status` agentId 不匹配时返回 403 `unauthorized_agent`
- [ ] `PATCH /api/agent/tasks/{id}/status` 非法 status 值返回 400 `invalid_status`
- [ ] `POST /api/agent/tasks/{id}/records` 成功返回 `{"recordId":N,"taskStatus":"completed"}`
- [ ] `go test ./internal/handler/... -v` 覆盖 agent 端点，含错误场景

## Implementation Notes

- `taskId` 路径参数在 claim/status/record 端点为数字 ID（`int64`），在 content 端点为字符串
- 使用 chi 路由时，content 端点路径可设为 `/api/agent/tasks/{key}/content` 以区分
- 请求体解析失败（JSON 格式错误）返回 400 `missing_field`
